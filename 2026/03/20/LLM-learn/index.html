LLM 相关基本概念

一些LLM相关的基础知识，非常基础，仅作为基本概念梳理

一、基础层

1. LLM (Large Language Model, 大语言模型)

基于海量文本训练，通过预测下一个 token 来生成内容的概率模型。
  • 能力边界： 擅长泛化任务（写作、翻译、总结），不擅长精确计算和实时信息（除非外挂工具）。
  • 幻觉 (Hallucination)： 模型会一本正经地胡说八道。对策： 需要 RAG 或人工校验。
  • 上下文窗口 (Context Window)： 模型一次能“记住”的最大字数（如 128K）。影响： 决定了你能喂给它多少文档做分析。

2. Token

LLM 处理文本的最小单位（不是字，也不是词，通常是半个词或一个偏旁）。
  • 通俗类比： 模型的“计费单位”和“消化单位”。就像吃饭按“口”算，不是按“碗”算。
  • 成本计算： API 调用费通常按 Input Token + Output Token 计费。设计产品时要考虑Token 经济性（比如压缩 Prompt 能省钱）。
  • 速度影响： 生成速度通常用 Tokens/s 衡量。

3. Transformer & Attention Mechanism (注意力机制)

现代 LLM 的基石架构。Attention 让模型在处理一个词时，能“关注”到句子中其他相关的词（无论距离多远）。
  • 通俗类比： 读长文章时，读到后面能自动回想前面的关键信息，而不是读了后面忘前面。
  • 并行计算： 解释了为什么大模型训练需要大量 GPU。
  • 长文本依赖： 注意力机制决定了模型理解长逻辑的能力，但也带来了计算量平方级增长的问题（引出了后面的稀疏注意力等技术）。

4. BERT vs. LLM (GPT 类)

BERT 是双向编码模型（擅长理解），GPT 是单向解码模型（擅长生成）。
  • BERT： 像做阅读理解题的高手，适合搜索、分类、情感分析（判别式）。
  • LLM (GPT)： 像写作文的高手，适合对话、创作、代码生成（生成式）。
    选型策略： 如果需求是“判断这句话是不是骂人”，用 BERT 类小模型（便宜、快、准）；如果需求是“写一封道歉信”，必须用 LLM。不要动不动就调大模型，合适才是最好的。

二、架构层：如何让模型更聪明？

应对幻觉，我们并非毫无手段——RAG、微调、提示词工程/引用溯源、置信度展示（技术架构、产品设计、运营迭代）

5.RAG (Retrieval-Augmented Generation, 检索增强生成)

在用户提问时，先去外部知识库（如公司文档）检索相关信息，再把“问题 + 资料”一起喂给 LLM 生成答案。

  • 通俗类比： 开卷考试。LLM 是考生，外部知识库是参考书。考生先查书，再结合书里的内容回答问题，减少瞎编。
  • 解决痛点： 解决知识过时、私有数据缺失、幻觉问题。
  • 核心难点： 检索的准确率（Recall）。如果查不到相关资料，LLM 还是会瞎编。
  • 应用场景： 企业知识库问答、法律/医疗咨询（需要依据）。

6. Fine-tuning (微调) vs. Prompt Engineering (提示词工程)

  • Prompt： 通过优化输入指令来引导模型，不改模型参数。
  • Fine-tuning： 用特定领域数据重新训练模型部分参数，让模型“学会”新风格或新知识。
  • 微调： 像送员工去进修，改变他的思维模式（成本高，周期长，但能掌握深层行业逻辑）。
    数据门槛： 微调需要高质量标注数据，这是 PM 需要协调资源的重点。

7. AI Agent (智能体)

能够自主感知、规划、使用工具并执行任务的 LLM 系统。
  • 核心组件： 规划 (Planning)、记忆 (Memory)、工具使用 (Tools)。
  • 应用趋势： 2025-2026 年的主流方向。从 Chatbot (问答) 转向 Copilot (辅助) 再到 Agent (自主执行)。

8. Vector Database (向量数据库)

专门存储“向量 embeddings”的数据库，用于语义检索。

  • 通俗类比： 传统的关键词搜索是“找完全一样的字”；向量搜索是“找意思相近的词”。比如搜“苹果”，能搜出“水果”甚至“手机”（取决于语境）。
    • RAG 的基石： 没有向量库，RAG 就无法进行语义匹配。
    • Embeddings： 把文字变成一串数字（向量），距离越近意思越像。

三、优化与性能层：怎么降本增效？

9. KV Cache

在生成过程中，缓存之前计算过的 Key 和 Value 矩阵，避免重复计算。

  • 价值： 极大提升生成速度，降低延迟 (Latency)。
  • 代价： 占用显存 (GPU Memory)。长上下文会导致显存爆炸，这是限制 Context Window 的主要瓶颈之一。

10. MoE (Mixture of Experts, 混合专家模型)

一个大模型里包含多个 “专家”子网络 ，每次只激活其中一部分来处理特定问题。

  • 通俗类比： 医院挂号。头痛挂神经科，脚痛挂骨科。不需要全科医生（全参数模型）每次都全体出动，谁擅长谁上。
  • 优势： 在参数量巨大的情况下，保持推理速度快、成本低（因为每次只用了部分参数）。
  • 代表模型： Mixtral, Grok, Google Gemini 等都在用。

11. Quantization (量化)

降低模型参数的精度（如从 16 位浮点数降到 4 位整数），以减小模型体积。
- 通俗类比： 把高清原图压缩成 JPG，画质稍微损失一点，但文件小了 4 倍，传输更快。

四、安全与评估层

12. RLHF (Reinforcement Learning from Human Feedback, 基于人类反馈的强化学习)

让人类对模型输出打分，训练一个奖励模型，引导 LLM 生成更符合人类价值观的内容。

  • 对齐 (Alignment)： 解决模型“有毒”、“偏见”问题的核心技术手段。
  • 成本： 需要大量高质量人工标注，非常贵。

13. Prompt Injection (提示词注入) & Jailbreak (越狱)

用户通过特殊的指令欺骗模型，绕过安全限制，让它输出有害内容。

  • 通俗类比： 骗守卫开门。不说“我要进去偷东西”，而说“我是维修工，老板让我进去修水管”（社会工程学攻击）。

14. Evaluation Metrics (评估指标)

怎么知道模型好不好？

  • 传统 NLP： 准确率、召回率、F1。
    GenAI 特有：
  • 忠实度 (Faithfulness)： 回答是否基于检索内容？
  • 相关性 (Relevance)： 回答是否解决了问题？

应用

  • Diffusion Model（扩散模型） 是一种生成式 AI 模型，它通过模拟“从噪声中逐渐还原出清晰图像”的过程，来生成高质量的图片、音频或视频。
    • 它是目前 Midjourney、Stable Diffusion、DALL-E 3 等主流绘图工具的底层核心技术。
    • 应用场景主要有文生图、文生视频、图像编辑、超分辨率等。

分析不同定位

目前国内大模型已形成明显的差异化竞争格局：

  • Kimi是长文本与深度阅读的专家；解决了传统模型“读了后面忘前面”的问题
  • DeepSeek是极致性价比与硬核推理的极客首选；让中小型企业也能用得起大模型。对于代码生成、数学解题、逻辑推理这类硬任务，DeepSeek 的表现往往优于很多闭源模型。
  • 千问是全栈能力与生态整合的企业级全能手；“云 + 端 + 场景”的闭环（B端）
  • 豆包则是面向大众用户的 AI 生活伴侣与流量入口。语音对话流畅度、多模态玩法。

分析单一产品

分析一款 AI 工具，比如 Cursor，你觉得它成功的原因是什么？

Cursor 的成功在于它没有把 AI 当作一个“聊天插件”塞进编辑器，而是以 AI 为核心原生重构了代码编辑的全流程，实现了从“辅助输入”到“自主代理（Agent）”的范式转移，完美解决了开发者对上下文缺失和信任成本的两大痛点。

  • GitHub Copilot 只是 VS Code 的一个插件，起初感知不到整个项目结构，交互也受限。（现在也可以联系上下文）
  • Cursor 的核心护城河不是模型（它用的是通用的 Claude/GPT-4），而是它处理私有代码上下文的能力。对于 B 端产品，RAG（检索增强生成）的精度直接决定用户体验。
